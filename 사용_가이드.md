> Claude Code에게 일을 시키기 전에 먼저 이 가이드를 읽어주세요.

## 2. 프로젝트 폴더에 배치하는 방법

GitHub 리포지토리 루트에서 다음 구조로 배치합니다.

```
fire-evacuation-system/
├── CLAUDE.md                ← 여기 (루트)
├── README.md                ← 여기 (루트)
└── docs/
    ├── manual_v2.md
    ├── coordinate_convention.md
    ├── interface_contracts.md
    ├── risk_indicators.md
    ├── decisions.md
    ├── lessons_learned.md
    └── task_request_template.md
```

**중요**: `CLAUDE.md`는 반드시 **프로젝트 루트**에 두세요. Claude Code는
작업 디렉토리에서 `CLAUDE.md` 파일을 자동으로 찾아 모든 세션에서 로드합니다.
다른 위치에 두면 자동 로드가 안 됩니다.

`README.md`도 루트에 두세요. GitHub에서 자동으로 보여주는 첫 페이지입니다.

나머지 7개 파일은 `docs/` 폴더 아래에 둡니다. Claude Code는 작업 시
필요한 docs만 명시적으로 읽습니다.

## 3. 첫 Claude Code 세션 — 스켈레톤 만들기

폴더 구조가 비어있는 상태에서 Claude Code에게 다음을 요청합니다.

### 명령

```bash
cd fire-evacuation-system   # 빈 디렉토리
claude   # 또는 사용 중인 Claude Code 진입 명령
```

### 첫 메시지 (이걸 통째로 붙여넣기)

```
이 디렉토리에 fire evacuation system 프로젝트의 스켈레톤을 생성해줘.

먼저 다음 파일들을 정확한 위치에 배치할 거야 (내용은 첨부할게):

1. CLAUDE.md (프로젝트 루트)
2. README.md (프로젝트 루트)
3. docs/manual_v2.md
4. docs/coordinate_convention.md
5. docs/interface_contracts.md
6. docs/risk_indicators.md
7. docs/decisions.md
8. docs/lessons_learned.md
9. docs/task_request_template.md

이 9개 파일을 정확히 그대로 저장해줘.
저장 후, CLAUDE.md를 읽고 그에 따라 다음을 만들어줘:

A. requirements.txt - PyTorch 2.0+, neuraloperator, fdsreader, NetworkX,
   scipy, pybullet, gym-pybullet-drones, wandb, pytest, pyyaml, click,
   h5py, matplotlib, numpy

B. .gitignore - Python, PyTorch checkpoint, data/raw, data/processed,
   wandb, .DS_Store, __pycache__, *.pyc, .venv, .vscode 등

C. pyproject.toml - 최소한의 setuptools 또는 hatch 설정

D. src/ 디렉토리 구조 - 각 서브디렉토리에 __init__.py만 있는 빈 스켈레톤
   (src/shared, src/data_pipeline, src/dataset, src/models, src/training,
    src/evaluation, src/risk_map, src/path_planning, src/integration,
    src/visualization)

E. 다음 파일들은 Tier 1 (실제 동작): src/shared/constants.py와
   src/shared/normalization.py만 실제 구현해줘. CLAUDE.md와
   docs/interface_contracts.md를 보고 정확하게 구현해.
   
   각각 if __name__ == '__main__'에 PASS/FAIL 출력하는 self-test 포함.

F. tests/test_constants.py와 tests/test_normalization.py만 실제 작성.
   pytest로 검증 가능해야 함.

G. configs/ 디렉토리에 빈 yaml 파일 6개 (building, data, conv_lstm,
   pi_fno, risk_map, path_planning) - 모두 minimal placeholder만.

H. 나머지 src/ 모듈들은 함수 시그니처와 docstring만 작성하고 본문은
   raise NotImplementedError("Week N: implement using ...")로.

검증:
- python -m src.shared.constants → PASS 출력
- python -m src.shared.normalization → PASS 출력
- pytest tests/ -v → 둘 다 통과
- tree . (또는 find . -type f) 출력으로 모든 파일 확인

검증 후 7번 항목까지 통과해야 "완료" 판정. 검증 실패 시 수정 후 재검증.

마지막으로 git commit -m "Initial skeleton with full context files" 추천.
```

위 메시지에 9개 파일 내용을 첨부 (마크다운 코드 블록으로) 또는 파일 자체를
업로드합니다. Claude Code는 그것들을 정확히 저장한 후 스켈레톤을 만듭니다.

## 4. 매 세션마다 작업을 시키는 방법

스켈레톤이 만들어진 후, 새로운 작업을 시킬 때마다 다음 패턴을 사용합니다.

### 4.1 작은 작업 (한 함수, 한 모듈)

`docs/task_request_template.md`의 8섹션 템플릿을 사용합니다. 예시는
`task_request_template.md` 안에 3개 들어있습니다 (FED 함수, ConvLSTM 학습
스크립트, 좌표 유틸).

### 4.2 의사결정이 필요한 작업

```
질문: <상황 설명>

옵션을 분석하고 추천해줘. 다음 컨텍스트를 참고해:
- CLAUDE.md (자동 로드됨)
- docs/decisions.md (지금까지의 결정 패턴)
- docs/<관련 문서>

옵션별 장단점, 추천 안, 그 이유를 정리해줘. 합의되면
docs/decisions.md에 D-NNN으로 기록.
```

### 4.3 컨텍스트 보강 작업 (새 정보 추가)

```
새로 발견한 사실: <사실>

다음에 추가해야 할 것 같아:
- docs/lessons_learned.md (버그라면)
- docs/decisions.md (결정이라면)
- CLAUDE.md (프로젝트 전반에 영향이라면)

각 파일 어디에 추가할지 제안하고, 정확한 텍스트를 써줘.
```

## 5. Claude Code가 헷갈릴 때 가이드

### 상황 1: "이 프로젝트가 뭐 하는 거야?"
→ Claude Code가 CLAUDE.md를 안 읽었을 수 있습니다. 강제로 읽게 하기:
```
먼저 CLAUDE.md를 읽고, 어떤 프로젝트인지 한 단락으로 요약해줘.
그리고 6개 핵심 가설이 무엇인지 알려줘.
```

### 상황 2: "이 인터페이스를 바꿔도 돼?"
→ Claude Code가 인터페이스를 임의 변경하려 함:
```
docs/interface_contracts.md를 읽어. 이 인터페이스는 stable이야.
변경하려면 docs/decisions.md에 D-NNN 새 항목으로 제안하고,
내 승인을 받은 후에만 변경해.
```

### 상황 3: "이건 처음 보는 버그인데"
→ 거의 항상 lessons_learned에 있습니다:
```
docs/lessons_learned.md L-001부터 L-011까지 모두 읽어보고,
지금 발생한 에러와 일치하는 게 있는지 확인해.
```

### 상황 4: "다음에 뭘 할지 모르겠어"
→ manual_v2와 현재 진행 상황 비교:
```
docs/manual_v2.md를 읽고 14주 일정 중 우리가 어디 있는지 파악해.
CLAUDE.md의 "Current Project State" 섹션도 봐.
다음 1-2일에 우선순위 가장 높은 작업 3개를 추천해.
```

## 6. 실수 방지 체크리스트

작업 결과를 받았을 때 다음을 확인하세요.

**파일 작성 후 즉시:**

- [ ] 파일이 의도한 경로에 정확히 생성됐는지
- [ ] `if __name__ == '__main__'` 블록이 PASS를 출력하는지
- [ ] import 에러가 없는지
- [ ] 인터페이스 계약(`interface_contracts.md`)과 일치하는지

**중요한 변경 후:**

- [ ] CLAUDE.md의 "Current Project State"가 업데이트됐는지
- [ ] 새 결정은 `decisions.md`에 D-NNN으로 추가됐는지
- [ ] 새 버그는 `lessons_learned.md`에 L-NNN으로 추가됐는지

**커밋 전:**

- [ ] pytest가 모두 통과하는지
- [ ] git diff로 의도 외 변경이 없는지
- [ ] 메시지가 명확한지 (`week6: implement fds_extractor` 같은)

## 7. 자주 마주칠 질문

### Q1. CLAUDE.md를 매번 첨부해야 하나요?
**A**: 아니요. Claude Code는 작업 디렉토리의 `CLAUDE.md`를 자동으로
읽습니다. 다른 docs/는 명시적으로 가리킬 때만 읽습니다.

### Q2. 9개 파일 중 어떤 게 가장 중요한가요?
**A**:
- `CLAUDE.md` (1순위, 매 세션 자동 로드)
- `interface_contracts.md` (2순위, 구현 시 매번 참고)
- `coordinate_convention.md` (3순위, 공간 관련 코드 시 매번 참고)
- 나머지는 필요할 때만

### Q3. 매뉴얼이 너무 길어요. 줄여야 하나요?
**A**: 아닙니다. `manual_v2.md`는 인덱스 역할이라 길어도 됩니다.
Claude Code는 명시적으로 읽으라고 할 때만 읽습니다 — 매 세션마다
자동 로드되지 않습니다. CLAUDE.md만 짧게 유지하면 됩니다.

### Q4. 팀원이 4명인데 모두가 Claude Code를 써도 되나요?
**A**: 네. 같은 컨텍스트 파일을 보기 때문에 일관됩니다. 다만
**`Current Project State`** (CLAUDE.md 마지막 섹션)는 한 사람이 갱신
책임을 맡아야 충돌이 없습니다.

### Q5. 컨텍스트가 너무 많아서 Claude Code가 느려져요.
**A**: 아닙니다. CLAUDE.md만 자동 로드되고 나머지는 필요할 때만
읽습니다. 자동 로드되는 것은 약 18KB로 매우 작습니다.

## 8. 단계별 사용 시나리오

### 시나리오 A: 처음부터 시작 (지금 상태)

1. 빈 GitHub 리포지토리 생성
2. 로컬에 clone
3. 위 9개 파일을 `CLAUDE.md`, `README.md`는 루트에, 나머지 7개는 `docs/`에 배치
4. Claude Code에게 [3장의 첫 메시지]를 보냄
5. 스켈레톤 생성 후 검증
6. `git commit -m "Initial skeleton"`
7. push

### 시나리오 B: Week 6 데이터 파이프라인 작업

1. `docs/manual_v2.md` "Phase B: Data Pipeline (Week 6)" 섹션 읽기
2. `docs/task_request_template.md`의 Example 3 (좌표 유틸) 같은 형식으로 작업 요청 작성
3. Claude Code에게 한 모듈씩 의뢰 (fds_extractor.py 먼저)
4. 검증 (`python -m src.data_pipeline.fds_extractor`)
5. 다음 모듈 (normalize.py)
6. 모두 끝나면 `Current Project State` 업데이트

### 시나리오 C: 새 버그 발견

1. 디버깅하면서 원인 파악
2. Claude Code에게:
   ```
   다음 버그를 docs/lessons_learned.md에 L-NNN으로 추가해줘:
   증상: ...
   원인: ...
   해결: ...
   ```
3. 추가된 내용 확인
4. 미래 같은 종류의 작업할 때 lessons_learned가 자동으로 컨텍스트가 됨

### 시나리오 D: 인터페이스 변경 필요

1. `docs/decisions.md`에 D-NNN으로 새 결정 제안:
   ```
   D-NNN: <변경 제안>
   Date: <today>
   Decision: <변경 내용>
   Alternatives: <고려한 대안>
   Rationale: <이유>
   ```
2. 팀원 합의 후 `docs/interface_contracts.md` 변경
3. 영향 받는 모든 코드 업데이트
4. 테스트 통과 확인 후 commit

## 9. 응급 상황 대응

### Claude Code가 엉뚱한 코드를 만듦
→ 즉시 중단하고 다음을 보내세요:
```
Stop. 다음을 다시 확인해:
- CLAUDE.md "Hard Constraints" 섹션
- docs/interface_contracts.md
- 우리가 합의한 격자 (60, 40, 6)와 단위 (m)

방금 만든 코드 중 이 제약을 어긴 부분이 있으면 모두 되돌려.
```

### 토큰 한계 도달
→ 새 세션 시작. CLAUDE.md가 자동 로드되므로 컨텍스트는 유지됩니다.
이전 세션의 마지막 결과만 새 세션에 짧게 알려주면 됩니다.

### 인터페이스 충돌
→ 한 모듈이 다른 모듈의 인터페이스를 위반:
```
방금 만든 X 모듈이 docs/interface_contracts.md의 Y 인터페이스를 위반했어.
인터페이스 변경은 D-NNN 결정 없이 못 해.
- X를 수정해서 인터페이스에 맞추거나
- D-NNN 결정 제안을 만들어 (그러면 내가 검토하고 결정함)
```
